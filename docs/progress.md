## Carryover notes
- **M4 decisions to remember:** the public view took the canonical path ŌĆö `GET /api/listings/{id}` is public (live only), and the owner's full view **moved** to `GET /api/my/listings/{id}`. `ListingPublic` is a **standalone** model, deliberately *not* a `ListingRead` subclass, so a private field added to the owner's view cannot join the public one. `status` is **not** on the public model (browse returns `live` only, so it would be a constant and a channel for a future state to leak); M12 may add a deliberate public "under offer" flag. Keyword search covers **public text only** ŌĆö reaching into `ListingPrivate` would make the search box an identity oracle.
- **Money range filters cast to NUMERIC.** The `Money` TypeDecorator stores Decimal as TEXT, so a bare SQL comparison is lexicographic (`'90000.00' > '200000.00'` is true as strings). Only the boundary comparison casts; stored and returned values stay exact Decimals. **On the Postgres swap this needs a look** ŌĆö `NUMERIC(14,2)` enforces precision there, and a value Pydantic's `max_digits=14` accepts could overflow at the DB layer (flagged by the appsec review, not a vulnerability today).
- **The M4 review found an over-claiming test, and it is the lesson worth keeping.** S9 claimed to prove "appearing in browse requires an admin approval" but walked only from `draft` ŌĆö and since `pause` requires `live`, it could never reach `pause ŌåÆ edit ŌåÆ resume`, *the exact bypass M3 found*. It was not vacuous (it still fails if browse drops its status filter) but it proved the easy half. **S10** now covers the republish corridor from an approved listing, checking after *every* step, and is verified by reverting the M3 guard. Generalization: *a reachability test is only as strong as the states it can actually reach ŌĆö check what its starting state makes impossible.*
- **`seed/seed.py` refuses any non-SQLite database**, guarded on the **session's bound engine** rather than global settings, so the check travels with the capability. The first fix guarded only `main()`, which left `seed(session)` open to every other caller ŌĆö caught by the bounded re-verification round, which is exactly why that round exists.
- **Deferred out of M4, with no milestone claimed:** user-selectable sorting and the "multiple" filter (the rest of FR-10), and analytics events ŌĆö there is **no `track()` wrapper in the codebase at all**, and no acceptance criterion covered analytics, so M4 emits none rather than shipping an untested side channel on the milestone that first exposes public data.
