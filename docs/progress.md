## Carryover notes
- **What the M5 build actually established (slices 1–7), worth keeping:**
  - **Slice 6 was one dependency swap.** Re-gating document downloads changed `get_owned_listing` → `require_private_access` and touched nothing in the body. That was the *stated test* of the design (plan.md: "if this slice needs new logic, the gate was built wrong"), and it is the concrete payoff of one-function-per-trust-boundary. M2's `test_listing_download.py` passes **unedited** — verified by diffing the file against `ecd8934`, not by assertion.
  - **The audit table earns its place only for values a later transition overwrites.** Criteria C9/C10 failed because the first implementation wrote an event row at request *creation*. `decided_at` gets overwritten by every decision; `created_at` never moves — so a `requested` event was a second copy of a fact that cannot drift. **Generalize: before adding an event row, ask what it preserves that the row itself loses.**
  - **`require_request_decider` returns 403 for a missing row as well as a foreign one** — deliberately the opposite of `get_owned_listing`'s 404-for-both. Both satisfy `security.md` §1.4's real rule (*be consistent within a boundary*) and both refuse to be existence oracles; they differ in which uniform answer is safe, because a listing's existence is a secret and an access-request id is not. Criteria S1 + S7 together pin this.
  - **D10 is verified, not assumed, and was rewritten.** Sabotaging the gate two ways (honour `revoked`; key on `listing.status`) makes it fail and *name* the paths. It was then rewritten from a depth-3 sequence product to a **BFS over observed states**: 77s → 30s, no depth limit, plus `test_d10b`, a depth-2 exhaustive backstop covering BFS's pruning assumption. Re-verified after the rewrite — 54 and 34 violations where the old version found 1 and 6. **If you touch the gate, re-run the sabotage; a reachability test nobody has seen fail proves nothing.**
- **M5 spec decisions (owner-approved 2026-07-20), all recorded in `specs/005-nda-gate/spec.md` § Decisions:** **D1** existence rule — a **never-published** listing returns **404** to a non-owner (still a secret), a **published** one returns **403** `nda_access_required` (the UI needs to distinguish "gone" from "ask for access"); **D2** an approved buyer **keeps** access when the listing leaves `live` — `revoke` is the only way back; **D3** decided requests are **terminal**, re-granting deferred to **(Post-MVP) on FR-13**; **D4** signing is idempotent and freezes `nda_version`; **D5** FR-14 ships its **profile half only** — no `verified` placeholder, M10 owns it; **D6** decisions write an append-only **`accessrequestevent`** table; **D7** the seller's queue is **`GET /api/my/listings/{id}/access-requests`**, guarded by the existing `get_owned_listing`.

- **M4 decisions to remember:** the public view took the canonical path — `GET /api/listings/{id}` is public (live only), and the owner's full view **moved** to `GET /api/my/listings/{id}`. `ListingPublic` is a **standalone** model, deliberately *not* a `ListingRead` subclass, so a private field added to the owner's view cannot join the public one. `status` is **not** on the public model (browse returns `live` only, so it would be a constant and a channel for a future state to leak); M12 may add a deliberate public "under offer" flag. Keyword search covers **public text only** — reaching into `ListingPrivate` would make the search box an identity oracle.
- **Money range filters cast to NUMERIC.** The `Money` TypeDecorator stores Decimal as TEXT, so a bare SQL comparison is lexicographic (`'90000.00' > '200000.00'` is true as strings). Only the boundary comparison casts; stored and returned values stay exact Decimals. **On the Postgres swap this needs a look** — `NUMERIC(14,2)` enforces precision there, and a value Pydantic's `max_digits=14` accepts could overflow at the DB layer (flagged by the appsec review, not a vulnerability today).
