## ▶ NEXT ACTION
**M6 — realtime chat**: **`/run-milestone chat`**

M6 is the first WebSocket surface. Its scope fold-ins (`docs/milestones.md` § Scope fold-ins → M6) are already written: a conversation **unique per (listing, buyer)**, `last_read_at` per participant for unread counts, a **WebSocket error contract** (close codes for auth-fail / non-member / revocation / rate-cap, landing as an `error_handling.md` addendum), and message events for the FR-16 email fallback that M8 delivers.

**M5 leaves M6 two things to honour.** First, `design_implementation.md` says approving access also creates the `conversation` row — that belongs to M6, with the conversation model. Second, and more important: **revocation must re-deny the socket and history immediately** (`security.md` §1.5). M5 made `revoked` a real terminal state; M6 is where a live connection has to notice it. The in-memory `{conversation_id: [sockets]}` registry is single-instance **by construction** and fails *silently* behind a load balancer, so keep the fan-out behind a `publish(conversation_id, message)` port from day one — persist → publish → fan out, in that order.

M6 is **not** on the security-critical list, but `scripts/check_appsec_trigger.py` will almost certainly escalate it: it adds routes, a state machine, and websockets. Let the diff decide, as the constitution's 2026-07-19 amendment intends.

## Carryover notes
- **What the M5 build actually established (slices 1–7), worth keeping:**
  - **Slice 6 was one dependency swap.** Re-gating document downloads changed `get_owned_listing` → `require_private_access` and touched nothing in the body. That was the *stated test* of the design (plan.md: "if this slice needs new logic, the gate was built wrong"), and it is the concrete payoff of one-function-per-trust-boundary. M2's `test_listing_download.py` passes **unedited** — verified by diffing the file against `ecd8934`, not by assertion.
  - **The audit table earns its place only for values a later transition overwrites.** Criteria C9/C10 failed because the first implementation wrote an event row at request *creation*. `decided_at` gets overwritten by every decision; `created_at` never moves — so a `requested` event was a second copy of a fact that cannot drift. **Generalize: before adding an event row, ask what it preserves that the row itself loses.**
  - **`require_request_decider` returns 403 for a missing row as well as a foreign one** — deliberately the opposite of `get_owned_listing`'s 404-for-both. Both satisfy `security.md` §1.4's real rule (*be consistent within a boundary*) and both refuse to be existence oracles; they differ in which uniform answer is safe, because a listing's existence is a secret and an access-request id is not. Criteria S1 + S7 together pin this.
  - **D10 is verified, not assumed, and was rewritten.** Sabotaging the gate two ways (honour `revoked`; key on `listing.status`) makes it fail and *name* the paths. It was then rewritten from a depth-3 sequence product to a **BFS over observed states**: 77s → 30s, no depth limit, plus `test_d10b`, a depth-2 exhaustive backstop covering BFS's pruning assumption. Re-verified after the rewrite — 54 and 34 violations where the old version found 1 and 6. **If you touch the gate, re-run the sabotage; a reachability test nobody has seen fail proves nothing.**
- **M5 spec decisions (owner-approved 2026-07-20), all recorded in `specs/005-nda-gate/spec.md` § Decisions:** **D1** existence rule — a **never-published** listing returns **404** to a non-owner (still a secret), a **published** one returns **403** `nda_access_required` (the UI needs to distinguish "gone" from "ask for access"); **D2** an approved buyer **keeps** access when the listing leaves `live` — `revoke` is the only way back; **D3** decided requests are **terminal**, re-granting deferred to **(Post-MVP) on FR-13**; **D4** signing is idempotent and freezes `nda_version`; **D5** FR-14 ships its **profile half only** — no `verified` placeholder, M10 owns it; **D6** decisions write an append-only **`accessrequestevent`** table; **D7** the seller's queue is **`GET /api/my/listings/{id}/access-requests`**, guarded by the existing `get_owned_listing`.
- **The revocation fold-in silently invalidated `security.md`'s audit claim** (corrected 2026-07-20). That file asserted access decisions were audited by `access_request.decided_at` — true when the only decisions were approve/deny, false once `revoke` made a row travel `requested → approved → revoked`, because a single `decided_at` holds only the **last** decision and revoking overwrote *when access was granted*. **The general rule recorded there: adding a transition to a state machine can invalidate an audit design that was correct for the old one — re-ask "what does this overwrite?" whenever a state machine grows.** Criterion C10 is the test that fails against the old design.
- **26 of M5's 60 criteria were invisible to CI's `spec-coverage` job** until fixed (2026-07-20). The exact set was **D2–D10, E2–E5, F3, all of S (S1–S8), all of X (X1–X4)**. *(An earlier version of this note said "the whole gate group D, all of S, every X" — wrong at both ends: **D1 matched fine**, and it omitted E2–E5 and F3 entirely. Corrected after the docs audit reconstructed the pre-fix spec and re-ran the actual regex. A note that gets the headline number right and the detail wrong is the kind of claim that survives review by sounding specific.)* Two causes: criteria written GIVEN→THEN with no explicit **WHEN**, and prose **inside** the bold id (`**S1 — title.**`) breaking the id capture, which needs `**S1**` closed immediately. The job would have reported full coverage while the crown-jewel tests went uncited — the exact failure it exists to prevent. **When writing a spec, run `python scripts/check_spec_coverage.py NNN` and check the criterion *count*, not just the exit code.**
- **A test that passes before implementation is unverified** — TDD's guarantee is "it failed first." Two M5 tests were caught passing vacuously (a 404 assertion satisfied by Starlette's *unmounted-route* 404 rather than by any gate logic); both were strengthened to assert the machine `code`, which the unmounted route has no way to produce. **The check that catches this class: after writing the red set, assert the number of passing new tests is zero — and for any that do pass, prove they are regression pins rather than vacuous** (E4/E5 are the legitimate case).
- **M4 decisions to remember:** the public view took the canonical path — `GET /api/listings/{id}` is public (live only), and the owner's full view **moved** to `GET /api/my/listings/{id}`. `ListingPublic` is a **standalone** model, deliberately *not* a `ListingRead` subclass, so a private field added to the owner's view cannot join the public one. `status` is **not** on the public model (browse returns `live` only, so it would be a constant and a channel for a future state to leak); M12 may add a deliberate public "under offer" flag. Keyword search covers **public text only** — reaching into `ListingPrivate` would make the search box an identity oracle.
- **Money range filters cast to NUMERIC.** The `Money` TypeDecorator stores Decimal as TEXT, so a bare SQL comparison is lexicographic (`'90000.00' > '200000.00'` is true as strings). Only the boundary comparison casts; stored and returned values stay exact Decimals. **On the Postgres swap this needs a look** — `NUMERIC(14,2)` enforces precision there, and a value Pydantic's `max_digits=14` accepts could overflow at the DB layer (flagged by the appsec review, not a vulnerability today).
- **The M4 review found an over-claiming test, and it is the lesson worth keeping.** S9 claimed to prove "appearing in browse requires an admin approval" but walked only from `draft` — and since `pause` requires `live`, it could never reach `pause → edit → resume`, *the exact bypass M3 found*. It was not vacuous (it still fails if browse drops its status filter) but it proved the easy half. **S10** now covers the republish corridor from an approved listing, checking after *every* step, and is verified by reverting the M3 guard. Generalization: *a reachability test is only as strong as the states it can actually reach — check what its starting state makes impossible.*
- **`seed/seed.py` refuses any non-SQLite database**, guarded on the **session's bound engine** rather than global settings, so the check travels with the capability. The first fix guarded only `main()`, which left `seed(session)` open to every other caller — caught by the bounded re-verification round, which is exactly why that round exists.
